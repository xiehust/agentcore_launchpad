# Research: strands_studio_ui canvas + code-generation architecture (for launchpad port)

- **Query**: How does strands_studio_ui build the visual canvas and generate Python Strands code, and what is the minimal set to port into agentcore_launchpad (canvas + property panel + code generation, NOT deploy)?
- **Scope**: internal (source = `/home/ubuntu/workspace/strands_ui`, the full clone; git HEAD `456a042 fix system prompt setting`)
- **Date**: 2026-07-11

> The vendored copy `agentcore_launchpad/vendor-src/strands_studio_ui` is at an older merge commit (`d56f396`) and differs from the clone. All anchors below are against the clone at `/home/ubuntu/workspace/strands_ui`.

---

## TL;DR for the port

1. **Code generation is 100% frontend and pure** — `src/lib/code-generator.ts`, `graph-code-generator.ts`, `graph-validator.ts`, `connection-validator.ts` import ONLY `@xyflow/react` types. Verified zero `fetch`/`axios`/`api-client`/`import.meta.env` in any canvas file. This logic ports verbatim.
2. **The contract**: `generateStrandsAgentCode(nodes, edges, graphMode) → { code, imports, errors }`. Final `.py` file = `result.imports.join('\n') + '\n\n' + result.code`. Single Python file. No requirements.txt is generated by the canvas (deps are static; see §3).
3. **Biggest obstacle = styling/stack clash**, not logic. strands_ui = React 19 + Tailwind CSS v4 + shadcn/ui + light theme. launchpad = React 18 + hand-written CSS token design system (`src/theme/*.css`, dark theme) + i18next + react-router, **no Tailwind at all**. Every node/panel in strands_ui is Tailwind-utility-classed and will not render correctly in launchpad without either (a) adding/scoping Tailwind or (b) rewriting to launchpad's `.panel/.btn/.field/.input/.chip` classes.
4. **Library React-compat is fine**: `@xyflow/react@^12` peer dep is `react >=17`; `@monaco-editor/react@^4.7` peer dep includes `^18`. Both work on launchpad's React 18 — no React upgrade required.
5. **Integration point already exists**: launchpad `frontend/src/pages/CreateAgent.tsx:11` defines `STUDIO_URL` (`VITE_STUDIO_URL`, default `http://localhost:5273`) and renders a "studio" method card (`CreateAgent.tsx:300-316`, className `studio-link`) that currently links OUT to the standalone studio. The port replaces that external link with an embedded canvas route/page.

## §1. Self-described architecture (from CLAUDE.md / README)

"Visual agent flow builder" → drag-drop nodes → generate Python using Strands Agent SDK → execute (sync or streaming). Frontend: React 19 + Vite + Tailwind v4 + shadcn/ui (New York) + `@xyflow/react` + Monaco + lucide. Backend (FastAPI) only does execution/persistence/deploy — irrelevant to the port. Two generation modes:
- **Sequential/orchestrator/swarm** (default) — `code-generator.ts`.
- **Graph Mode** (DAG) — toggled in flow-editor top-right; routes to `graph-code-generator.ts` via `generateStrandsAgentCode(...graphMode=true)`.

Key documented rules (CLAUDE.md "Critical Architecture Rules"): each MCP node connects to exactly one agent; `callback_handler=None` on every Agent to prevent streaming duplication; MCP clients used inside `with` context managers; conversation via `--messages` (priority) or `--user-input` CLI args.

## §2. Canvas architecture (React Flow / @xyflow/react)

### State management (`flow-editor.tsx`)
- `FlowEditor` supports **controlled** (external `nodes/edges/onNodesChange/onEdgesChange`) or **uncontrolled** (`useNodesState`/`useEdgesState`) modes. `main-layout.tsx` uses controlled: it owns `useState<Node[]>`/`useState<Edge[]>` and `graphMode` (`main-layout.tsx:50-53`).
- Controlled change handlers manually translate React Flow change events (`position`/`select`/`remove`) into new arrays (`flow-editor.tsx:76-136`) and cascade edge removal when a node is deleted (`:103-107`).
- `nodeTypes` registered (`flow-editor.tsx:38-47`): `agent`, `orchestrator-agent`, `swarm`, `tool`, `mcp-tool`, `input`, `output`, `custom-tool`. **`graph-builder` is NOT registered** (GraphBuilderNode exists + has a property renderer but is not droppable via the palette; Graph Mode is a toggle, not a container node).
- Drag-drop: palette sets `dataTransfer.setData('application/reactflow', type)` (`node-palette.tsx:79`); `onDrop` (`flow-editor.tsx:177-239`) creates `{ id: type_Date.now(), type, position, data }` and injects default data for `agent` (`:200-210`) and `mcp-tool` (`:213-227`). **Other node types drop with only `{ label: "<type> node" }`** — their defaults come from destructuring fallbacks in each node/property renderer.
- Connect: `onConnect` validates via `isValidConnection()` then `addEdge` (`flow-editor.tsx:139-151`); also passes `isValidConnection` to React Flow for live handle validation (`:153-159`, `:271`).
- Canvas chrome: `<Controls/> <MiniMap/> <Background variant=Dots/>`, `deleteKeyCode=[Delete,Backspace]`, `multiSelectionKeyCode=[Meta,Ctrl]`, `fitView`. Requires `import '@xyflow/react/dist/style.css'` (`flow-editor.tsx:20`).

### Handles per node (source of the connection contract)

| Node | Target handles (id) | Source handles (id) |
|---|---|---|
| `agent` (`agent-node.tsx`) | `user-input` (top), `tools` (left), `orchestrator-input` (left-lower) | `output` (bottom) |
| `orchestrator-agent` (`orchestrator-agent-node.tsx`) | `user-input`, `tools`, `orchestrator-input` | `sub-agents` (right), `output` (bottom) |
| `swarm` (`swarm-node.tsx`) | `user-input` (top) | `sub-agents` (right), `output` (bottom) |
| `tool` (`tool-node.tsx`) | — | `tool-output` (right) |
| `custom-tool` (`custom-tool-node.tsx`) | `config` (left, unused in gen) | `tool-output` (right) |
| `mcp-tool` (`mcp-tool-node.tsx`) | — | `mcp-tools` (right) |
| `input` (`input-node.tsx`) | — | `output` (right) |
| `output` (`output-node.tsx`) | `input` (left) | — |

### Connection rules (`connection-validator.ts:13-232`, enforced by `isValidConnection` `:335-495`)
- input.output → agent/orchestrator/swarm.`user-input`
- tool.`tool-output` / custom-tool.`tool-output` → agent/orchestrator.`tools`
- mcp-tool.`mcp-tools` → agent/orchestrator.`tools`
- agent/orchestrator/swarm.`output` → output.`input`
- orchestrator.`sub-agents` → agent.`orchestrator-input` (sub-agent as tool) and → orchestrator.`orchestrator-input` (hierarchical)
- swarm.`sub-agents` → agent.`orchestrator-input` (swarm membership)
- **Graph Mode extra rules**: any executable.`output` → any executable.`user-input` (agent↔orchestrator↔swarm) = DAG dependency edges (`:155-231`).
- Extra guards: no self-connections; tools cannot target swarm (`:400-406`); orchestrator-hierarchy cycle guard `wouldCreateCircularDependency` (`:409-414`); Graph-Mode cycle guard via `detectCycles` (`:439-461`); **MCP node → only one agent** (`:464-477`); no duplicate edges (`:479-492`). Invalid connect shows a browser `alert()` (`flow-editor.tsx:147`) — the port should replace this with launchpad's toast (`useToast`).

### shadcn/radix primitives actually present
`src/components/ui/` has `button.tsx`, `input.tsx`, `label.tsx`, `select.tsx`, `tabs.tsx`. **The node components and property-panel do NOT use these** — they hand-roll `<div>`/`<select>`/`<input>` with Tailwind classes. So the port does not need to bring shadcn/radix unless you want the `ui/` primitives. `base-node.tsx` is a shadcn helper that no node file imports.

## §3. Code-generation contract (the deliverable)

### Entry & output shape
```ts
generateStrandsAgentCode(nodes: Node[], edges: Edge[], graphMode = false)
  : { code: string; imports: string[]; errors: string[] }
```
- Consumers assemble the file as: `result.imports.join('\n') + '\n\n' + result.code` (`code-panel.tsx:24`, `main-layout.tsx:272-273`).
- **Single Python file** (`strands_agent.py`), downloaded as a Blob (`code-panel.tsx:67-77`). No multi-file output. **No requirements.txt generated** — imports are a fixed set (see below); dependency install is the deploy pipeline's job (launchpad's).
- `errors[]` non-empty ⇒ `code=''` (validation failed). Errors cover: missing agent/input/output nodes, unconnected input/output, swarm with no agents (`code-generator.ts:73-118`). Graph mode delegates validation to `validateGraphStructure` (`graph-code-generator.ts:261-266`).

### Static imports always emitted (`code-generator.ts:30-38`)
```
from strands import Agent, tool
from strands.models import BedrockModel
from strands_tools import calculator, file_read, shell, current_time, http_request, editor, retrieve, mem0_memory
import json, os, asyncio, argparse
```
Conditionally added: `from strands.tools.mcp import MCPClient` + `mcp` transports if any `mcp-tool` (`:42-47`); `from strands.multiagent import Swarm` if any swarm (`:50-53`); `from strands.models.openai import OpenAIModel` if any agent uses `modelProvider==='OpenAI'` (`:67-71`). Graph mode adds `from strands.multiagent import GraphBuilder` (`graph-code-generator.ts:248`) and a trimmed strands_tools import (`:249` — only calculator/file_read/shell/current_time).

### Node types the generator supports
- **agent** — `BedrockModel` or `OpenAIModel`; system prompt; temperature/maxTokens; optional extended thinking (`additional_request_fields.thinking` for Bedrock, `reasoning_effort` for OpenAI); `callback_handler=None`. Generated as global instance OR (if MCP-connected) created inside `main()`'s `with` block (`generateAgentCode` `:269`, `generateAgentModelOnly` `:236`).
- **orchestrator-agent** — sub-agents become `@tool`-decorated functions (`generateAgentAsToolCode` `:995`); orchestrator gets `tools=[...subFunctions]`; supports hierarchical orchestrators (`generateOrchestratorAsToolCode` `:1093`) and `coordinationPrompt` appended to system prompt.
- **swarm** — `Swarm([agents], entry_point=, max_handoffs, max_iterations, execution_timeout, node_timeout, repetitive_handoff_*)` (`generateSwarmCode` `:1499`); swarm-member agents get `name=` (`generateSwarmAgentCode` `:318`).
- **graph (Graph Mode)** — `GraphBuilder().add_node/add_edge/set_entry_point/build()`; edges = executable.output→executable.user-input; entry points from `validateGraphStructure().entryPoints` (`graph-code-generator.ts:354-405`).
- **tool** (built-in) — name mapped to `strands_tools` symbol. Map (`code-generator.ts:443-453`): calculator, file_read (+ alias file_reader), shell, current_time, http_request, editor, retrieve, mem0_memory. **Note mismatch**: property-panel offers `file_write` (`property-panel.tsx:434`) but the map has no `file_write` → falls back to `calculator`. Graph generator's map is smaller (only calc/file_read/shell/current_time, `graph-code-generator.ts:113-119`).
- **custom-tool** — user Python pasted verbatim, wrapped with `@tool` (`generateCustomToolCode` `:1449`); function name extracted via regex `def <name>(` (`:901-909`).
- **mcp-tool** — `MCPClient` with stdio / streamable_http / sse transport; `startup_timeout` from node `timeout`; tools pulled via `list_tools_sync()` inside `with` context in `main()` (`generateMCPSetupCode` `:368`, main assembly `:554-720`).
- **input / output** — structural only (mandatory, define execution agent + validation); no code object. Execution agent chosen by tracing input→agent edges (`findConnectedAgent` `:495-516`).

### Model providers
Two: **AWS Bedrock** (default, `BedrockModel`) and **OpenAI** (`OpenAIModel` with `client_args.base_url` → supports **OpenAI-compatible endpoints**; API key via `os.environ["OPENAI_API_KEY"]`, never hardcoded). Anthropic is commented out in the agent panel but present in the orchestrator panel dropdown (`property-panel.tsx:181` vs `:721`) — no dedicated Anthropic codegen branch exists, so it would fall through to the Bedrock branch. Bedrock model catalog (`property-panel.tsx:55-136`) includes Claude 4.5 Haiku/Sonnet, Claude 4 Sonnet, Claude 3.7 Sonnet (global/us/eu/apac variants), GPT-OSS-120B, Qwen3, DeepSeek-V3.1, Amazon Nova. **Default modelId** used across generators: `us.anthropic.claude-3-7-sonnet-20250219-v1:0`.

### Main-execution assembly (`generateMainExecutionCode` `:518-899`)
Builds `async def main(user_input_arg, messages_arg)` with `--messages` > `--user-input` > default priority; branches for swarm (`result.status`/`node_history`), streaming (`agent.stream_async`, only if `data.streaming` AND an output node is connected), and sync (`agent(user_input)`); wraps in `with <mcp clients>:` when MCP tools reach the execution agent; ends with `argparse` + `asyncio.run(main(...))`. This is the exact code launchpad's deploy pipeline would consume as "agent source code."

## §7. Dependencies & the styling/stack clash (the real work)

### strands_ui canvas-only deps (`package.json`)
Runtime needed for canvas+panel+codegen: `@xyflow/react@^12.8.4`, `@monaco-editor/react@^4.7.0` (+ `monaco-editor` peer), `lucide-react@^0.543.0`, `react-syntax-highlighter@^15.6.6` (only used by `custom-tool-node.tsx` preview — droppable). Tailwind-only: `clsx`, `tailwind-merge`, `class-variance-authority`, `tailwindcss@^4`, `@tailwindcss/vite`, `tailwindcss-animate`, `tw-animate-css`, `radix-ui`/`@radix-ui/*` (only if adopting `ui/` primitives).

### launchpad frontend (`frontend/package.json`)
React **18.3**, Vite **6**, react-router-dom 6, i18next. **No Tailwind, no clsx, no xyflow, no monaco, no lucide.** Styling = hand-written CSS in `src/theme/tokens.css` (design tokens: `--bg:#0B0E0D` dark theme, `--panel`, `--ink`, `--amber`, series colors, `--mono`/`--sans` fonts) + `src/theme/app.css` (419 lines of component classes).

### Compatibility verdict
- **React**: xyflow v12 peer `react >=17`, monaco-react peer includes `^18` → **both run on React 18; no upgrade needed.** (Don't bump launchpad to React 19.)
- **Vite**: `@xyflow/react` and monaco work on Vite 6; the `@tailwindcss/vite` plugin is Tailwind-specific and would only be added if you choose to bring Tailwind.
- **Styling (the clash)**: strands_ui nodes/panels are ~100% Tailwind utility classes on a **light** theme (`bg-white`, `border-gray-200`, `bg-blue-50`, etc.). launchpad is a **dark** token theme with semantic classes. Three options:
  1. **Rewrite node/panel markup to launchpad classes** (`.panel`, `.btn`/`.btn.primary`, `.field`, `.input`/`.input.mono`, `.chip` + color variants `good/warn/crit/muted/amber/blue/aqua/llm/tool/mem/gw`, `.note`, `.cfg-grid`, `.selchip`, `.method` — all in `theme/app.css`). Most consistent with launchpad, most hand-work. Keep the pure lib/* files untouched; only the `.tsx` presentational layer changes.
  2. **Add Tailwind v4 scoped to the canvas route** (e.g. a prefix or a scoped root). Fastest to get parity with the source, but Tailwind's preflight/reset can bleed into launchpad's global CSS; light-theme node colors will clash with the dark shell unless overridden.
  3. **Hybrid**: keep React Flow's own CSS + minimal utility shim for the nodes, restyle only the palette/property panel to launchpad classes.
- **Icons**: add `lucide-react` (launchpad has none) or swap to whatever icon set launchpad uses.
- **`cn()`**: only needed if keeping Tailwind; drop `src/lib/utils.ts` under option 1.

### Backend-coupling verification
`grep` for `fetch|axios|api-client|import.meta.env|VITE_` across flow-editor, node-palette, property-panel, base-node, code-panel, all `nodes/*`, and all four lib generator/validator files → **zero hits**. The port is therefore low-risk on logic and concentrated almost entirely in the styling/stack-integration layer.

## Caveats / Not Found
- `file_write` built-in tool is selectable in the UI but unmapped in the generator (silently becomes `calculator`). Decide whether to add it during the port.
- The Bedrock model list is hardcoded in `property-panel.tsx`; launchpad may want to source models from its own registry/api instead.
- `graph-builder` node type has UI/property support but is not droppable (Graph Mode is a canvas-level toggle). Do not expose it as a palette item unless you also register it in `nodeTypes`.
- Node `data` defaults are split between drop-time defaults (agent, mcp-tool only) and destructuring fallbacks everywhere else — when porting, ensure defaults survive so the generator's `= fallback` values match the UI's displayed values.
- I did not deep-read `execution-panel.tsx`/`api-client.ts` internals (out of scope); their existence and endpoints are documented from CLAUDE.md + imports only.
